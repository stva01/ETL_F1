Prologue: The Philosophy of Distributed Compute

Before writing a single line of code, you must fundamentally change how you think about data. When you write Python with Pandas, you are a chef in a kitchen: you pick up a carrot, you chop it, you put it in a pot. The data is small enough to fit in your hands (your computer's RAM).

When you write PySpark, you are the head chef of a massive factory. You cannot touch the carrots. There are millions of them, spread across 50 different conveyor belts (Executors). You can only write instructions on a whiteboard (Transformations), and at the very end, ring a bell to start the factory (Actions).

⚡ Lazy Evaluation & The DAG

When you write transformations in PySpark (filter, join, withColumn), Spark does absolutely nothing. It doesn't read the data from S3. It doesn't load it into memory. It simply draws a logical map of what it wants to do. This map is called a DAG (Directed Acyclic Graph).

Spark only actually goes to S3, reads the data, and executes the math when you call an Action (like .count(), .show(), .write(), or .isEmpty()).

Why does it do this? Because Spark's brain (the Catalyst Optimizer) looks at your entire DAG and rewrites it to be more efficient before a single byte of data is processed. If you filter for "year == 2024" and then select "driver_name", Catalyst is smart enough to realize it only needs to read the "year" and "driver_name" columns from S3, completely ignoring the "lap_times" column. If you did this line-by-line in Pandas, it would read everything.

Deep Dive: Read the Apache Spark SQL Guide on how the Catalyst Optimizer parses your code. Understanding Catalyst is what separates junior data engineers from senior ones.

❓ Interview Question: What is the difference between a Transformation and an Action? Transformations are lazy (e.g., select, filter, join); they return a new DataFrame and build the DAG. Actions are eager (e.g., count, write, collect); they trigger the execution of the DAG and return a result to the driver.

Module 1: The Engine Architecture (Driver vs. Executor)

When you run code in AWS Glue, you aren't running it on your laptop. You are renting a cluster of machines. Understanding how they talk to each other is critical to writing code that doesn't crash.

The Muscle] end D -- "Sends Tasks & Plan" --> E1 D -- "Sends Tasks & Plan" --> E2 D -- "Sends Tasks & Plan" --> E3 E1 -- "Returns Results" --> D E2 -- "Returns Results" --> D E3 -- "Returns Results" --> D style D fill:#d29922,stroke:#000,color:#000 style E1 fill:#58a6ff,stroke:#000,color:#000 style E2 fill:#58a6ff,stroke:#000,color:#000 style E3 fill:#58a6ff,stroke:#000,color:#000

The Driver holds the SparkContext and SparkSession. It is the orchestrator. It figures out the execution plan, splits the work into "tasks", and hands them out. The Executors are the workers. They hold chunks of your data (called Partitions) in their RAM, perform the actual filtering/math, and send the final computed results back to the Driver.

The Driver is the architect holding the blueprint. The Executors are the construction workers holding the bricks. If the architect tries to hold all the bricks in their office (by calling .collect() on a massive dataset), the architect's office will collapse (Out of Memory error).

In our Glue setup code, we initialize this connection:

# This code simply connects your "Brain" to AWS's "Muscle"
sc          = SparkContext.getOrCreate()
glueContext = GlueContext(sc)
spark       = glueContext.spark_session # Your steering wheel

# The try/except is purely an AWS Glue Studio notebook quirk.
# When you click "Run" interactively, AWS doesn't pass --JOB_NAME.
# When scheduled as a job, it does. This handles both gracefully.
job = Job(glueContext)
try:
    args = getResolvedOptions(sys.argv, ["JOB_NAME"])
    job.init(args["JOB_NAME"], args)
except Exception:
    job.init("interactive-notebook", {})
❓ Interview Question: What is a partition in Spark, and why does it matter? A partition is a logical chunk of a dataset. If you read a 10GB file with 5 executors, Spark splits it into 5 partitions (2GB each). Each executor works on 1 partition independently. If you have 10,000 tiny files, Spark might create 10,000 partitions, which causes massive overhead managing tasks. (This leads directly into Module 2).

Module 2: I/O, Formats, and The Small File Problem

Data engineering at its core is moving data from a "Read-Optimized" format to a "Query-Optimized" format. APIs give you JSON. Data warehouses need Parquet.

JSON vs. Parquet: A Tale of Two Layouts

JSON is row-based. All data for Max Verstappen (name, lap time, speed) is stored together in one big block of text. Parquet is columnar. It stores all the names together, then all the lap times together, then all the speeds together.

graph TD subgraph "Row-Based (JSON) - Read Everything" R1["{'driver': 'Max', 'time': '82.1', 'speed': '310'}"] R2["{'driver': 'Lewis', 'time': '83.2', 'speed': '305'}"] end subgraph "Columnar (Parquet) - Read Only What You Need" C1["Driver Block: [Max, Lewis]"] C2["Time Block: [82.1, 83.2]"] C3["Speed Block: [310, 305]"] end R1 -.-> C1 R2 -.-> C1 R1 -.-> C2 R2 -.-> C2 R1 -.-> C3 R2 -.-> C3 style R1 fill:#f85149,stroke:#000,color:#000 style R2 fill:#f85149,stroke:#000,color:#000 style C1 fill:#3fb950,stroke:#000,color:#000 style C2 fill:#3fb950,stroke:#000,color:#000 style C3 fill:#3fb950,stroke:#000,color:#000

If dbt asks Athena to run SELECT driver FROM table, using JSON forces S3 to scan the entire file to find the driver names. Using Parquet allows S3 to literally skip the "Time" and "Speed" blocks entirely. This is called Column Pruning, and it makes Parquet incredibly fast and cheap to query.

Deep Dive: Databricks excellent breakdown of Parquet internal architecture.

The Multi-Line JSON Trap (Jolpica Specific)

Standard Spark assumes a JSON file is "JSON Lines" format—one distinct object per line. But the Ergast/Jolpica API returns a single, massive, beautifully formatted JSON object that spans hundreds of lines.

If you read this with standard spark.read.json(), Spark will try to parse the first line ({"MRData":), fail because it's incomplete, and either crash or return garbage data.

# The magic switch for Jolpica/Ergast data
# WARNING: This forces Spark to read the ENTIRE file into a SINGLE partition.
# This is okay for Jolpica (files are small), but would crash a 50GB file.
df_jolpica = spark.read.json("s3://.../results.json", multiLine=True)

🔥 The Small File Problem (Crucial Architecture Concept)

This is the most common mistake junior engineers make in the cloud. They write a loop that processes data one race at a time, appending to S3:

# BAD PATTERN (What we initially did)
for race in races:
    df.write.mode("append").parquet("s3://bucket/results/")

This results in an S3 bucket with 20,000 files, each 5KB in size.

Imagine S3 is a library. Asking the librarian to fetch 1 book of 500 pages is fast. Asking the librarian to fetch 5,000 post-it notes scattered across 5,000 different shelves will take hours, even though it's the exact same amount of text.

Technically, when Athena runs a query, it calls the S3 ListObjects API to find the files. S3 limits this to 1,000 requests per second. If you have 100,000 files, just listing them takes 100 seconds—before Athena has read a single byte of data.

# THE FIX: .coalesce(1)
# This forces the Executors to hand all their data back to the Driver, 
# which squishes it into exactly ONE file per partition before writing.
df_clean.coalesce(1).write.mode("overwrite").partitionBy("year").parquet(path)

Deep Dive: Databricks guide to the Small File Problem.

❓ Interview Question: What is the difference between coalesce() and repartition()? Both reduce partitions, but their mechanics are entirely different. repartition() forces a full network shuffle to ensure data is perfectly balanced across nodes. coalesce() just merges existing partitions locally on the same node without moving data over the network. For fixing the small file problem at write time, always use coalesce(1) because it's instantaneous. Use repartition("column") when you need to fix data skew *before* a heavy join.

Module 3: DataFrame Transformations & Immutability

In Pandas, you can do df['col'] = 5 and mutate the dataframe in place. In PySpark, DataFrames are strictly immutable. When you call withColumn, Spark doesn't change the original DataFrame; it creates a brand new DataFrame that points back to the original data plus your new instruction in the DAG.

Immutability is required for distributed fault tolerance. If an Executor dies while processing data, Spark cannot "roll back" a changed file. Instead, it just spins up a new Executor and reruns the exact same immutable transformation steps from the original source data.

Type Casting: Schema-on-Read vs Schema-on-Write

Databases like Postgres enforce a schema before you insert data (Schema-on-Write). Spark/Parquet do not. You can write whatever you want. But if you write "44" as a string, and later try to SUM() it in dbt, it will fail. You must explicitly cast types during your cleaning phase.

from pyspark.sql.types import IntegerType, DoubleType, LongType

# LongType is for massive numbers (milliseconds since epoch)
# IntegerType maxes out at ~2.1 billion. LongType goes to 9 quintillion.
df = df.withColumn("driver_number", F.col("driver_number").cast(IntegerType()))
df = df.withColumn("time_millis", F.col("time_millis").cast(LongType()))

Literals & Math

# F.lit() injects a hardcoded value into every row.
# Why not just write "kaggle"? Because Spark needs it wrapped as a Column object for the JVM.
df = df.withColumn("source", F.lit("kaggle"))

# Math operations to convert OpenF1 seconds to milliseconds
df.withColumn("pit_duration_ms", 
    (F.col("pit_duration").cast(DoubleType()) * 1000).cast(LongType())
)

Deduplication

# Drop rows that are 100% identical across ALL columns
df.dropDuplicates()

# Drop rows that have the same driver_id, keeping the first one it finds
df.dropDuplicates(["driver_id"])

Module 4: Taming Nested JSON (The Jolpica Magic)

This is advanced PySpark. Most tutorials stop at flat CSVs. Jolpica data from the Ergast API looks like Russian nesting dolls: MRData -> RaceTable -> Races[] -> Results[] -> Driver.

When Spark reads this with multiLine=True, it infers a complex schema of Structs (dictionaries) and Arrays (lists).

graph TD A["Root: MRData (Struct)"] --> B["RaceTable (Struct)"] B --> C["Races (Array of Structs)"] C --> D["Race 1 (Struct)"] C --> E["Race 2 (Struct)"] D --> F["Results (Array of Structs)"] F --> G["Driver 1 Result (Struct)"] F --> H["Driver 2 Result (Struct)"] G --> I["Driver Object (Struct): {name, code}"] style C fill:#d29922,stroke:#000,color:#000 style F fill:#f85149,stroke:#000,color:#000 style I fill:#58a6ff,stroke:#000,color:#000

Step 1: Exploding Arrays Vertically

F.explode() is the magic key. It takes an array column and creates a new row for every item inside it. It "flattens" the array vertically.

# Races is an Array of Structs. Explode makes every race its own row.
races = df_raw.select(F.explode("MRData.RaceTable.Races").alias("race"))

# Now 'race' is a Struct. But inside 'race' is ANOTHER array (Results). 
# We must chain another explode!
results = races.select(
    F.lit(2024).alias("season"), # We add context because explosion loses it
    F.explode("race.Results").alias("r")
)
# Now 'r' represents ONE single driver's result for ONE single race.

Step 2: Digging into Structs with Dot Notation

Once exploded, the data is a flat table of Structs. You use dot notation to drill down.

flat = results.select(
    F.col("r.Driver.driverId").alias("driver_id"),
    F.col("r.Constructor.name").alias("team_name")
)

Step 3: Array Indexing (The Tricky Part)

What if a struct contains an array? E.g., a driver drove for two teams in one year (like Pierre Gasly in 2023), so Constructors is an Array of Structs. We use brackets [0] to grab the first one, treating the data like a Python dictionary.

# Grab index 0 of the Constructors array, then grab its 'constructorId' key
F.col("r.Constructors")[0]["constructorId"].alias("constructor_id")

The Entity Resolution Join

In our bridge_sources.py, we used the Kaggle drivers table as a "bridge" to translate IDs.

# We need to translate Kaggle's integer 'driverId' into Jolpica's string 'driver_ref'
lookup = driver_lookup.select("kaggle_driver_id", "driver_ref")

# Why 'left' join? Because if a driver is missing from the lookup, 
# we still want to KEEP the Kaggle result, just with a null driver_ref.
joined_df = kaggle_results.join(
    lookup,
    on=(kaggle_results["driverId"] == lookup["kaggle_driver_id"]),
    how="left"
).drop("kaggle_driver_id") # Drop the key so it doesn't duplicate in the output table

The Union (Stacking History and Present)

Unions stack rows on top of each other. But they are incredibly strict. Column names and data types must match exactly.

# unionByName is safer than union() because it matches by column name, not position.
# allowMissingColumns=True is a lifesaver: if Kaggle has 'status_id' but Jolpica 
# doesn't, Jolpica rows just get NULL for that column instead of throwing an error.
combined = kaggle_norm.unionByName(jolpica_norm, allowMissingColumns=True)
❓ Interview Question: What happens during a Spark Shuffle? When you join or groupBy, data with the same key must end up on the same Executor. Spark has to physically move data across the network between executors. This network I/O is called a Shuffle, and it is the #1 cause of slow Spark jobs. Never join on highly ununique columns (like join on status where 90% of rows say "Finished").

Module 6: Window Functions

Window functions are the "boss level" of SQL and Spark. They allow you to run GROUP BY logic without collapsing the rows.

The Scenario

We stacked Kaggle 2024 data and Jolpica 2025 data using a Union. But what if the ingestion script accidentally scraped 2024 from Jolpica too? Now we have two rows for Max Verstappen in 2024 Race 1. We need to group by (Year, Round, Driver), rank the sources (Jolpica=1, Kaggle=2), and keep only rank 1.

If we used a normal GROUP BY, we would lose all the individual race data! Window functions solve this.
graph TD A["Row 1: Max, 2024, Kaggle"] --> W[Window Group] B["Row 2: Max, 2024, Jolpica"] --> W C["Row 3: Lewis, 2024, Kaggle"] --> W2[Window Group] W -->|row_number assigns Rank 2| D((Drop)) W -->|row_number assigns Rank 1| E((Keep)) W2 -->|row_number assigns Rank 1| F((Keep)) style D fill:#f85149,stroke:#000,color:#000 style E fill:#3fb950,stroke:#000,color:#000 style F fill:#3fb950,stroke:#000,color:#000 
from pyspark.sql.window import Window

# 1. Define the window: "Group by these 3 columns, and sort them by this logic"
dedup_window = Window.partitionBy("year", "round", "driver_ref") \
                   .orderBy(
                       F.when(F.col("source") == "jolpica", 1)
                        .when(F.col("source") == "kaggle", 2)
                   )

# 2. Apply row_number() over that window. This adds a new column '_rank'
combined = combined.withColumn("_rank", F.row_number().over(dedup_window))

# 3. Filter to keep only the winners of the rank
combined = combined.filter(F.col("_rank") == 1).drop("_rank")

Deep Dive: Databricks Window Function overview.

❓ Interview Question: What is the difference between row_number(), rank(), and dense_rank()? If three people tie for 1st place: row_number() assigns 1, 2, 3 randomly. rank() assigns 1, 1, 1, then skips to 4. dense_rank() assigns 1, 1, 1, then goes to 2. For deduplication, always use row_number() because it guarantees no ties, meaning your filter (== 1) will strictly return exactly one row per group.

Module 7: UDFs (User Defined Functions)

⚠️ The Golden Rule of Spark

Spark is written in Scala/Java and runs on the JVM (Java Virtual Machine). When you use built-in functions like F.upper() or F.cast(), you are using ultra-fast, optimized Scala code.

When you write a Python def, Spark has to serialize the data from the JVM, send it out to a separate Python process, run your code, and serialize it back to the JVM. This serialization bounce destroys performance.

Only use UDFs if Spark's built-in functions physically cannot do the job.

We had to use a UDF in clean_jolpica.py to parse "1:27.452" into milliseconds because PySpark has no built-in "convert F1 lap time string to integer" function.

sequenceDiagram participant Executor as Scala Executor (Fast) participant Python as Python Worker (Slow) Executor->>Python: Serialize & Send Row 1 ("1:27.452") Python->>Python: Python splits strings, does math Python->>Executor: Serialize & Send back 87452 Executor->>Python: Serialize & Send Row 2 ("1:21.000") Note over Executor,Python: Doing this for 10 million rows = Massive Bottleneck 
# @F.udf tells Spark: "Treat this Python function as a Spark Column operation"
# LongType() tells Spark what data type the function will return
@F.udf(LongType())
def parse_lap_time_ms(time_str):
    if time_str is None: return None
    try:
        if ":" in time_str:
            parts = time_str.split(":")
            minutes = int(parts[0])
            sec_parts = parts[1].split(".")
            seconds = int(sec_parts[0])
            millis = int(sec_parts[1]) if len(sec_parts) > 1 else 0
            return minutes * 60_000 + seconds * 1_000 + millis
    except Exception:
        return None

# Usage: Pass the COLUMN, not the value. Spark handles the row-by-row iteration.
df = df.withColumn("lap_time_ms", parse_lap_time_ms(F.col("time_text")))

Module 8: Execution Traps (The "Gotchas")

Because of Lazy Evaluation, seemingly simple Python operations behave bizarrely in Spark.

The Empty Data Trap

How do you check if a DataFrame has data?

  • df.count() == 0: TERRIBLE. This triggers a full distributed scan of the entire dataset across all executors, just to count to 0. If you have 100 million rows, this takes minutes.
  • not df.head(1): BUG. In Python, head(1) returns a list like [Row(...)]. In Python, a list with items inside it is "Truthy" (evaluates to True). So not [Row(...)] evaluates to False. This if-statement will NEVER trigger, even if the dataframe is empty!
  • df.rdd.isEmpty(): PERFECT. This asks the executors to scan data, but the exact microsecond they find *one* row, they stop and return False. If the file is empty, it returns True instantly.
# The correct, safe way to check for empty data
df_raw = spark.read.json(path)
if df_raw.rdd.isEmpty():
    return None
