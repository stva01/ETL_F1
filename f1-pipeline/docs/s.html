⚡ Technology Stack

  • Storage: AWS S3 (Bronze/Silver raw files) · Delta Lake or Iceberg format on Parquet
  • Processing: PySpark on EMR / Databricks for initial load & incremental merges
  • Transformation: dbt Core / Cloud — staging → core → marts, full lineage graph
  • Warehouse: Snowflake / BigQuery / Redshift (star schema in Gold layer)
  • Orchestration: Apache Airflow or Prefect — DAG triggers on S3 event after each race
  • AI / MCP: Anthropic Claude + MCP server reading semantic tables for natural-language insights
02
Design Principles
Decisions that drive every table definition below

💸 Cost Reduction Strategy

  • Partition by race_year + round: Queries for a single season scan only that partition — not 75 years of data.
  • Cluster fact tables by driver_key and constructor_key — most dashboard queries filter on one of these.
  • Materialise marts as physical tables, not views — the dashboard hits pre-computed numbers, never re-aggregating 1M+ lap rows.
  • fact_lap_time is the largest table (billions of rows) — partition by year and cluster by race_key. Keep a separate fact_lap_summary mart for typical queries.
  • Cold storage for raw S3 files: S3 Glacier after 30 days; only Gold/Platinum tables stay in the hot warehouse.

📐 Star Schema Conventions

  • Surrogate keys on all dims (BIGINT auto-increment). Never join on business keys in prod queries.
  • Conformed dimensions: dim_driver is shared across facts. One source of truth.
  • SCD Type 2 on dim_constructor (team name/nationality can change — see Benetton→Renault→Lotus→Renault).
  • Grain is explicit: Every fact table has a grain comment in DDL and in the dbt model YAML.
  • No NULLs in FK columns — use a "Unknown" surrogate key (−1) for late-arriving dims.
  • Immutable facts: Historical race results never change. Only standings facts may be overwritten.

🔄 Incremental Load Pattern

  • After each race: S3 drop triggers Airflow DAG → PySpark reads new CSVs → MERGE into staging tables (upsert on natural key).
  • dbt runs in --select tag:incremental mode — only rebuilds models downstream of changed source tables.
  • Fact tables use dbt incremental materialisation with unique_key = result_id. New rows append; corrected rows merge.
  • Marts rebuild fully after each race batch (cheap — they're aggregations of bounded-size data).

🤖 MCP / AI Design Constraints

  • Every mart table has a human-readable description in the semantic_catalog table.
  • The mart_ai_context wide table is a single-row-per-driver denormalised view — designed to fit in an LLM context window for "tell me about Hamilton" type queries.
  • A nl_prompt_store table maps common question patterns to pre-built SQL, reducing AI hallucination on schema details.
  • All column names are fully spelled out (no abbreviations) so the MCP layer can describe them without a separate lookup.

dbt Model SQL
Key model definitions — staging, dimension, and fact layers
-- models/staging/stg_results.sql -- Staging model: clean raw results, cast types, add audit cols {{ config(materialized='view', schema='staging', tags=['staging']) }} WITH source AS ( SELECT * FROM {{ source('raw', 'results') }} ), renamed AS ( SELECT CAST(resultId AS INT) AS result_id, CAST(raceId AS INT) AS race_id, CAST(driverId AS INT) AS driver_id, CAST(constructorId AS INT) AS constructor_id, NULLIF(CAST(grid AS INT), 0) AS grid_position, NULLIF(CAST(position AS INT), 0) AS finish_position, COALESCE(CAST(points AS DECIMAL(5,2)), 0) AS points, CAST(laps AS INT) AS laps_completed, CAST(statusId AS INT) AS status_id, -- parse lap time string "1:21.546" → milliseconds {{ ms_to_laptime('fastestLapTime') }} AS fastest_lap_time_ms, NULLIF(CAST(fastestLapSpeed AS DECIMAL(6,3)), 0) AS fastest_lap_speed_kph, NULLIF(CAST(rank AS INT), 0) AS fastest_lap_rank, 'results.csv' AS _source_file, CURRENT_TIMESTAMP AS _loaded_at FROM source WHERE resultId IS NOT NULL ) SELECT * FROM renamed
-- models/core/dimensions/dim_driver.sql {{ config(materialized='table', schema='core', tags=['dim']) }} WITH drivers AS ( SELECT * FROM {{ ref('stg_drivers') }} ), debut_years AS ( SELECT r.driver_id, MIN(ra.year) AS debut_year, MAX(ra.year) AS final_year FROM {{ ref('stg_results') }} r JOIN {{ ref('stg_races') }} ra ON r.race_id = ra.race_id GROUP BY 1 ) SELECT {{ generate_surrogate_key(['d.driver_id']) }} AS driver_key, d.driver_id, d.driver_ref, d.code AS driver_code, CONCAT(d.forename, ' ', d.surname) AS full_name, d.forename, d.surname, d.date_of_birth, d.nationality, {{ nationality_to_code('d.nationality') }} AS nationality_code, {{ flag_emoji('d.nationality') }} AS flag_emoji, d.number AS permanent_number, dy.debut_year, dy.final_year, dy.final_year >= YEAR(CURRENT_DATE()) - 2 AS is_active, d.wikipedia_url, CURRENT_TIMESTAMP AS dw_created_at, CURRENT_TIMESTAMP AS dw_updated_at FROM drivers d LEFT JOIN debut_years dy ON d.driver_id = dy.driver_id
-- models/core/facts/fact_race_result.sql -- Incremental model: merge on result_key. Partition by season_year. {{ config( materialized = 'incremental', unique_key = 'result_key', schema = 'core', partition_by = {'field': 'season_year', 'data_type': 'int'}, cluster_by = ['driver_key', 'constructor_key'], tags = ['fact', 'incremental'] ) }} WITH results AS ( SELECT * FROM {{ ref('stg_results') }} {{- if is_incremental() }} WHERE _loaded_at > (SELECT MAX(dw_inserted_at) FROM {{ this }}) {{- endif }} ) SELECT {{ generate_surrogate_key(['res.result_id']) }} AS result_key, dr.race_key, dd.driver_key, dc.constructor_key, dr.circuit_key, ds.status_key, ra.year AS season_year, res.grid_position, res.finish_position, res.points AS points_scored, res.laps_completed, res.fastest_lap_time_ms, res.fastest_lap_speed_kph, res.fastest_lap_rank, res.fastest_lap_rank = 1 AS is_fastest_lap_point, res.finish_position = 1 AS is_winner, res.finish_position <= 3 AS is_podium, res.points > 0 AS is_points_finish, res.grid_position - res.finish_position AS positions_gained, FALSE AS is_sprint, CURRENT_TIMESTAMP AS dw_inserted_at FROM results res JOIN {{ ref('dim_race') }} dr ON res.race_id = dr.race_id JOIN {{ ref('dim_driver') }} dd ON res.driver_id = dd.driver_id JOIN {{ ref('dim_constructor') }} dc ON res.constructor_id = dc.constructor_id AND dc.scd_is_current = TRUE JOIN {{ ref('dim_race_status') }} ds ON res.status_id = ds.status_id JOIN {{ ref('stg_races') }} ra ON res.race_id = ra.race_id

⚠️ The Golden Rule

The fact_lap_time table will reach ~10 billion rows by 2030. Never let the dashboard query it directly. All lap-time queries go through mart_driver_season or aggregated materialized views. Direct lap queries are only for data engineering and deep analysis use-cases.

💡 BigQuery Optimisations

  • Use INTEGER RANGE partitioning on season_year (1950–2030, interval=1).
  • Enable column-level stats on driver_key and constructor_key for partition pruning hints.
  • Set table expiry on staging tables: 7 days. They're re-loadable from S3.
  • Flat-rate pricing for the dashboard (BI Engine reservation) — marts are queried constantly.
  • Use materialized views with enable_refresh=true for the top-5 dashboard queries.

❄️ Snowflake Optimisations

  • Use CLUSTER BY (season_year, driver_key) on all main fact tables.
  • Suspend warehouse after 1 minute idle — dashboard queries are sub-second on marts.
  • Use SEARCH OPTIMIZATION on dim_driver.full_name for AI/MCP free-text lookups.
  • Store mart tables as Transient (no fail-safe) — they rebuild from facts on every run.
  • Use Multi-cluster warehouses only during race-day batch loads.

Post-Race Ingestion Flow
What happens in the hours after each Grand Prix
## Airflow DAG: f1_post_race_pipeline ## Trigger: S3 PutObject event on s3://f1-raw/live-feed/YYYY/race_NNN/ ## Target SLA: Dashboard updated within 3 hours of chequered flag DAG: f1_post_race_pipeline schedule: S3 event trigger (EventBridge → SNS → Airflow API) concurrency: 1 # Only one race at a time 1. s3_sensor ↳ Watches s3://f1-raw/live-feed/{year}/race_{round}/complete.flag ↳ Timeout: 6 hours post-race 2. pyspark_ingest [EMR Serverless / Databricks Job] ↳ Read new CSV files from S3 Bronze partition ↳ Schema validation (Great Expectations) ↳ MERGE into Silver staging tables (upsert on natural key) ↳ Write audit record to pipeline_log 3. dbt_run_incremental ↳ dbt run --select tag:incremental --vars '{"race_id": NNN}' ↳ Builds: fact_race_result, fact_qualifying, fact_pit_stop, fact_lap_time ↳ Builds: fact_driver_standing, fact_constructor_standing 4. dbt_run_marts ↳ dbt run --select marts.* ↳ Full refresh of all 9 mart tables (~2 min total) 5. dbt_test ↳ dbt test --select marts.* ↳ Assert: no null FKs, standings are monotonic, grain is unique 6. invalidate_cache ↳ POST /api/cache/invalidate on dashboard service ↳ Notify: Slack #f1-data-ops channel 7. mcp_context_refresh ↳ Rebuild mart_ai_context for drivers with new data ↳ Update mcp_query_log with race metadata Total pipeline time: ~25–40 minutes end-to-end

✅ Data Quality Gates

  • Row count check: New race must have 16–24 result rows (modern grid size). Pipeline fails if outside range.
  • Points validation: Sum of points in a race must match expected total for the era (e.g., top-10 points system post-2010).
  • Referential integrity: Every result's driver_id and constructor_id must exist in dim tables before fact load.
  • Duplicate guard: MERGE on (race_id, driver_id) prevents double-loading the same race.
  • Freshness alert: If staging tables haven't received new rows within 12 hours of race day, PagerDuty alert fires.
